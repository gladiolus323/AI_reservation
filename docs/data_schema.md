## 개요

이 문서는 병원 검사 예약 시스템의 데이터 구조를 정의합니다.

### 예약일시 vs 실제시작시간

- **예약일시**: 병원에서 환자에게 안내하는 예약 시간
- **실제시작시간**: 검사가 실제로 시작되는 시간

같은 환자가 같은 예약시간에 여러 검사를 받는 경우:
- 검사들은 순차적으로 진행됨
- 실제시작시간은 이전 검사 종료 후로 계산됨

예시:
```
예약일시: 2026-03-19 10:45 (4개 검사)
├─ 검사1: 실제시작 10:45, 소요 15분 → 종료 11:00
├─ 검사2: 실제시작 11:00, 소요 10분 → 종료 11:10
├─ 검사3: 실제시작 11:10, 소요 15분 → 종료 11:25
└─ 검사4: 실제시작 11:25, 소요 10분 → 종료 11:35
```

### 날짜/시간 형식
- **예약일시**: `YYYY-MM-DD HH:mm` (예: `2025-09-20 10:30`)
- **실제시작시간**: `YYYY-MM-DD HH:mm` (예: `2025-09-20 10:45`)
- **가장빠른외래예약일자**: `YYYY-MM-DD` (예: `2025-10-15`)

### 규칙 (Rules) 상세

규칙은 JSON 배열 형식으로 저장되며, 각 규칙은 다음 구조를 가집니다.

```json
[
  {
    "avoid_tags": ["#Nuclear"],
    "gap_days": 1,
    "reason": "방사성 동위원소 간섭 방지"
  }
]
```

#### 규칙 필드

| 필드 | 타입 | 설명 |
|------|------|------|
| avoid_tags | String[] | 피해야 할 태그 목록 |
| gap_days | Integer | 필요한 최소 간격 (일) |
| reason | String | 규칙 사유 |

#### 현재 적용된 규칙

| 선행 태그 | 피해야 할 태그 | 간격 | 사유 | 적용 검사 수 |
|-----------|---------------|------|------|-------------|
| `#Nuclear` | `#Nuclear` | 1일 | 방사성 동위원소 간섭 방지 | 11개 |
| `#Barium` | `#Abdomen` | 3일 | 바륨 잔류로 인한 영상 품질 저하 | 1개 |
| `#Sedation` | `#Treadmill` | 1일 | 진정 후 협조 곤란 | 4개 |

## 4. 파일 간 관계

```
source.xlsx (원본)
     │
     ▼
patient_schedule.csv
     │
     │ 처방코드 참조
     ▼
constraints.csv
     │
     │ 장비유형/소요시간 참조
     ▼
schedule.html (시각화)
```

### 참조 관계

1. **patient_schedule.처방코드** → **constraints.처방코드**
   - 모든 예약의 처방코드는 constraints에 존재해야 함

2. **constraints.장비유형** → **schedule.html 리소스 배정**
   - 시각화 시 장비유형별로 검사실 그룹화 및 배정

## 5. 데이터 검증 규칙

### 필수 검증

1. **처방코드 존재 여부**: patient_schedule의 모든 처방코드가 constraints에 존재
2. **장비유형 매핑**: constraints의 모든 장비유형이 유효함

### 형식 검증

1. **날짜 형식**: `YYYY-MM-DD` 또는 `YYYY-MM-DD HH:mm`
2. **시간 형식**: `HH:mm` (10분 단위)
3. **JSON 형식**: 태그와 규칙은 유효한 JSON 배열

### 연속 검사 검증

1. **순차 배치**: 같은 환자+예약시간+장비유형의 검사는 순차적으로 배치
2. **시간 연속성**: 이전 검사 종료 시간 = 다음 검사 시작 시간

### 규칙 검증

1. **검사간 규칙**: 같은 환자의 예약이 규칙을 위반하지 않음
   - 예: 핵의학 검사 후 1일 이내 다른 핵의학 검사 불가

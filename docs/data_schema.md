# 데이터 스키마 문서

## 개요

이 문서는 병원 검사 예약 시스템의 데이터 구조를 정의합니다.

---

## 데이터 파이프라인 구조

```
┌─────────────────────────────────────────────────────────────────────────┐
│                          데이터 흐름도                                    │
└─────────────────────────────────────────────────────────────────────────┘

                         ┌──────────────┐
                         │  source.xlsx │  ← 원본 데이터 (병원 실제 데이터)
                         │   (원본)      │
                         └──────┬───────┘
                                │
              ┌─────────────────┼─────────────────┐
              │                 │                 │
              ▼                 ▼                 ▼
    ┌─────────────────┐ ┌────────────────┐ ┌───────────────┐
    │ patient_schedule│ │resource_schedule│ │  constraints  │
    │     .csv        │ │     .csv       │ │     .csv      │
    │   (가공 데이터)  │ │  (합성 데이터)  │ │ (가공+합성)   │
    └────────┬────────┘ └───────┬────────┘ └───────┬───────┘
             │                  │                  │
             └──────────────────┼──────────────────┘
                                │
                                ▼
                    ┌───────────────────────┐
                    │   AI 예약 에이전트     │ ← MCP 도구로 데이터 조회
                    │   (Claude + Tools)    │
                    └───────────┬───────────┘
                                │
                                │   ← 병원 정책 적용
                                ▼
                    ┌───────────────────────┐
                    │   hospital_rules.txt  │  ← 설정 파일 (병원별 커스텀)
                    │      (설정)           │
                    └───────────────────────┘
```

### 파일 유형 분류

| 파일 | 유형 | 설명 |
|------|------|------|
| source.xlsx | 원본 데이터 | 병원 예약 시스템에서 추출한 실제 데이터 |
| patient_schedule.csv | 가공 데이터 | source.xlsx에서 환자 예약 정보만 추출 |
| resource_schedule.csv | 합성 데이터 | source.xlsx 기반 장비/리소스 배정 정보 생성 |
| constraints.csv | 가공+합성 | 처방 목록(가공) + 태그/규칙(합성) |
| hospital_rules.txt | 설정 파일 | 병원별 커스텀 가능한 예약 정책 |

## 1. source.xlsx (원본 데이터)

병원 예약 시스템에서 추출한 원본 데이터입니다. 모든 가공/합성 데이터의 기반이 됩니다.

### 용도
- 환자 예약 정보 원천
- 처방코드 및 처방명 목록
- 실제 예약 패턴 분석 기반

### 주요 컬럼 (예상)
| 컬럼명 | 설명 |
|--------|------|
| 처방ID | 고유 처방 식별자 |
| 환자번호 | 환자 고유 번호 |
| 처방코드 | 처방 고유 코드 |
| 처방명 | 처방 명칭 |
| 예약일시 | 예약 날짜 및 시간 |
| 외래예약일자 | 외래 진료 예약일 |

### 파생 데이터
- `patient_schedule.csv`: 환자별 예약 정보 추출
- `resource_schedule.csv`: 장비 배정 정보 생성
- `constraints.csv`: 처방 목록 추출 후 태그/규칙 추가

## 2. patient_schedule.csv (가공 데이터)

환자 예약 정보를 저장하는 파일입니다. source.xlsx에서 필요한 컬럼만 추출하여 생성합니다.

### 스키마

| 컬럼명 | 타입 | 필수 | 설명 | 예시 |
|--------|------|------|------|------|
| 처방ID | String | Y | 고유 처방 식별자 | `173871269~1` |
| 환자번호 | String | Y | 환자 고유 번호 | `10009105` |
| 처방분류 | String | Y | 처방 분류 (진료과) | `영상의학과(CT)`, `신경과` |
| 처방코드 | String | Y | 처방 고유 코드 | `RC900002`, `NM010001` |
| 처방명 | String | Y | 처방 명칭 | `(검진)Chest CT(CE)` |
| 환자수진부서 | String | N | 환자 수진 부서 (약어) | `IMC`, `NS`, `EN` |
| 예약일시 | DateTime | Y | 예약 날짜 및 시간 | `2025-09-20 10:30` |
| 실제시작시간 | DateTime | Y | 검사 실제 시작 시간 | `2025-09-20 10:30` |
| 외래예약총건수 | Integer | N | 외래 예약 총 건수 | `5` |
| 외래예약건수최근1년 | Integer | N | 최근 1년 외래 예약 건수 | `3` |
| 가장빠른외래예약일자 | Date | N | 가장 빠른 외래 예약 날짜 | `2025-10-15` |

### 예약일시 vs 실제시작시간

- **예약일시**: 병원에서 환자에게 안내하는 예약 시간
- **실제시작시간**: 검사가 실제로 시작되는 시간

같은 환자가 같은 예약시간에 여러 검사를 받는 경우:
- 검사들은 순차적으로 진행됨
- 실제시작시간은 이전 검사 종료 후로 계산됨

예시:
```
예약일시: 2026-03-19 10:45 (4개 검사)
├─ 검사1: 실제시작 10:45, 소요 15분 → 종료 11:00
├─ 검사2: 실제시작 11:00, 소요 10분 → 종료 11:10
├─ 검사3: 실제시작 11:10, 소요 15분 → 종료 11:25
└─ 검사4: 실제시작 11:25, 소요 10분 → 종료 11:35
```

### 날짜/시간 형식
- **예약일시**: `YYYY-MM-DD HH:mm` (예: `2025-09-20 10:30`)
- **실제시작시간**: `YYYY-MM-DD HH:mm` (예: `2025-09-20 10:45`)
- **가장빠른외래예약일자**: `YYYY-MM-DD` (예: `2025-10-15`, 비어있을 수 있음)

### 환자수진부서 약어

| 약어 | 부서명 |
|------|--------|
| IMC | 심장혈관내과 |
| NS | 신경외과 |
| EN | 이비인후과 |
| NP | 신경과 |
| IMG | 영상의학과 |

### 데이터 통계
- 총 건수: 약 7,900건
- 기간: 2025-08-01 ~ 2026-09-19
- 고유 환자 수: 약 2,000명
- 고유 처방코드 수: 250개

## 3. resource_schedule.csv (합성 데이터)

장비/리소스별 예약 현황을 저장하는 파일입니다. source.xlsx 기반으로 리소스 배정 로직을 적용하여 생성한 합성 데이터입니다.

### 스키마

| 컬럼명 | 타입 | 필수 | 설명 | 예시 |
|--------|------|------|------|------|
| 처방ID | String | Y | 고유 처방 식별자 (FK → patient_schedule) | `173164247~1` |
| 리소스ID | String | Y | 장비/리소스 고유 ID | `R001`, `R010` |
| 리소스명 | String | Y | 장비/리소스 이름 | `CT_1`, `청력검사실_1` |
| 장비유형 | String | Y | 장비 유형 | `CT`, `MRI`, `청력검사실` |
| 예약일시 | DateTime | Y | 예약 날짜 및 시간 | `2025-09-04 16:00` |
| 실제시작시간 | DateTime | Y | 검사 실제 시작 시간 | `2025-09-04 16:00` |
| 종료시간 | DateTime | Y | 검사 종료 시간 | `2025-09-04 16:20` |

### 리소스ID 체계

| 리소스ID 범위 | 장비유형 |
|--------------|----------|
| R001~R005 | CT |
| R006~R009 | MRI |
| R010~R013 | 청력검사실 |
| R014~R016 | 초음파(US) |
| R017~R019 | SPECT/Gamma Camera |

| R024~R025 | 신경심리검사실 |
| R026~R027 | 심초음파실 |
| ... | ... |

### 용도
- AI 에이전트의 `get_unavailable_slots` 도구에서 사용
- 특정 장비유형의 예약 현황 조회
- 예약 가능 시간대 계산

### 예시 데이터
```csv
처방ID,리소스ID,리소스명,장비유형,예약일시,실제시작시간,종료시간
173164247~1,R001,CT_1,CT,2025-09-04 16:00,2025-09-04 16:00,2025-09-04 16:20

```

## 4. constraints.csv (가공+합성 데이터)

검사 제약조건 및 태그 정보를 저장하는 파일입니다.
- **가공 부분**: 처방코드, 처방명 (source.xlsx에서 추출)
- **합성 부분**: 장비유형, 소요시간, 태그, 규칙 (임상 기준으로 정의)

### 스키마

| 컬럼명 | 타입 | 필수 | 설명 | 예시 |
|--------|------|------|------|------|
| 처방코드 | String | Y | 처방 고유 코드 (PK) | `RC900002` |
| 처방명 | String | Y | 처방 명칭 | `(검진)Chest CT(CE)` |
| 장비유형 | String | Y | 필요 장비 유형 | `CT`, `MRI` |
| 소요시간 | Integer | Y | 검사 소요 시간 (분) | `20`, `30`, `45` |
| 태그 | JSON Array | Y | 검사 특성 태그 목록 | `["#CT","#Contrast"]` |
| 규칙 | JSON Array | Y | 검사간 제약 규칙 목록 | (아래 상세) |

### 태그 (Tags) 상세

태그는 JSON 배열 형식으로 저장됩니다.

```json
["#CT", "#Contrast", "#Abdomen", "#Fasting"]
```

#### 태그 종류

| 카테고리 | 태그 | 설명 |
|----------|------|------|
| **장비** | `#CT` | CT 촬영 |
| | `#MRI` | MRI 촬영 |
| | `#PET` | PET-CT |
| | `#SPECT` | SPECT/Gamma Camera |
| | `#US` | 초음파 |
| | `#Endoscopy` | 내시경 |
| **물질** | `#Nuclear` | 방사성 동위원소 사용 |
| | `#Contrast` | 조영제 사용 |
| | `#Barium` | 바륨 사용 |
| **상태** | `#Sedation` | 진정/수면 필요 |
| | `#Fasting` | 금식 필요 |
| **부위** | `#Brain` | 뇌/두부 |
| | `#Abdomen` | 복부 |
| | `#Chest` | 흉부 |
| | `#Spine` | 척추 |
| | `#Cardiac` | 심장 |

### 규칙 (Rules) 상세

규칙은 CSV 내 특수 형식으로 저장됩니다.

```
[{avoid_tags:[#Nuclear],gap_days:1,reason:방사성 동위원소 간섭 방지}]
```

#### 규칙 필드

| 필드 | 타입 | 설명 |
|------|------|------|
| avoid_tags | String[] | 피해야 할 태그 목록 |
| gap_days | Integer | 필요한 최소 간격 (일) - N일 후부터 가능 |
| reason | String | 규칙 사유 |

#### 현재 적용된 규칙

| 선행 태그 | 피해야 할 태그 | 간격 | 사유 | 적용 검사 수 |
|-----------|---------------|------|------|-------------|
| `#Nuclear` | `#Nuclear` | 1일 | 방사성 동위원소 간섭 방지 | 11개 |
| `#Barium` | `#Abdomen` | 3일 | 바륨 잔류로 인한 영상 품질 저하 | 1개 |
| `#Sedation` | `#Treadmill` | 1일 | 진정 후 협조 곤란 | 4개 |

### 소요시간 기준

소요시간은 임상학적 실제 검사 시간을 기준으로 설정됩니다.

#### 주요 소요시간

| 장비유형 | 일반 소요시간 | 비고 |
|----------|--------------|------|
| CT | 10~20분 | Temporal bone: 10분 |
| MRI | 30~45분 | L spine/Knee: 30분 |
| 초음파(US) | 5~20분 | 일반: 5분, 정밀: 20분 |
| 내시경실 | 10~30분 | 수면내시경: 30분 |
| 청력검사실 | 5~15분 | BERA: 10분 |
| PET-CT | 60분 | - |
| 신경심리검사실 | 10~60분 | 종합검사: 60분 |

### 데이터 통계
- 총 처방 수: 250개
- 태그 있는 처방: 190개 (76%)
- 규칙 있는 처방: 16개 (6.4%)

### 예시 데이터
```csv
처방코드,처방명,장비유형,소요시간,태그,규칙
HA443,경부제한적CT,CT,20,"[""#CT""]",[]
NM010001,Thyroid uptake scan 99mTc,SPECT/Gamma Camera,30,"[""#SPECT"",""#Nuclear""]","[{""avoid_tags"":[""#Nuclear""],""gap_days"":1,""reason"":""방사성 동위원소 간섭 방지""}]"
RU010001,Abdomen(간.담낭.담도.비장.췌장)-일반,초음파(US),5,"[""#US"",""#Abdomen"",""#Fasting""]",[]
```

## 5. hospital_rules.txt (설정 파일)

병원별로 커스텀할 수 있는 예약 정책 파일입니다. AI 에이전트의 시스템 프롬프트에 포함되어 추천 로직에 반영됩니다.

### 파일 형식

INI 스타일의 섹션 기반 텍스트 파일

### 섹션 구조

#### [병원 설정]
```
추천 기간: 30일 이내
```
- **추천 기간**: AI가 예약을 추천하는 최대 기간

#### [추천 규칙]
```
검사는 외래 진료일 이전에 완료될 수 있도록 추천해주세요
금식이 필요한 검사는 반드시 오전 10시 이전으로 추천해주세요 (필수)
같은 날 검사는 최대 2개까지만 추천해주세요
```
- 일반적인 예약 추천 규칙
- AI 에이전트가 추천 시 준수해야 할 지침

#### [우선순위]
```
1. 금식 검사는 무조건 오전 10시 이전에만 추천해야 합니다 (환자 선호보다 우선)
2. 검사 간 의료적 간격 규칙 준수가 중요합니다
3. 외래 진료 전 검사 완료를 고려하세요
4. 환자의 선호 시간대는 의료 규칙을 만족하는 범위 내에서만 반영해주세요
```
- 규칙 간 충돌 시 우선순위 정의
- 의료적 규칙 > 환자 선호

#### [금식 검사 특별 규칙]
```
- 금식이 필요한 검사: 환자가 오후를 원해도 반드시 오전 10시 이전으로 추천
- 금식 검사에서 환자가 "오후"를 선호한다고 해도, 의료적 이유로 오전만 가능함을 안내
- 예시 응답: "금식이 필요한 검사이므로 오전 10시 이전에만 예약이 가능합니다."
```
- `#Fasting` 태그가 있는 검사에 대한 특별 규칙

### 전체 예시
```
[병원 설정]
추천 기간: 30일 이내

[추천 규칙]
검사는 외래 진료일 이전에 완료될 수 있도록 추천해주세요
금식이 필요한 검사는 반드시 오전 10시 이전으로 추천해주세요 (필수)
같은 날 검사는 최대 2개까지만 추천해주세요

[우선순위]
1. 금식 검사는 무조건 오전 10시 이전에만 추천해야 합니다 (환자 선호보다 우선)
2. 검사 간 의료적 간격 규칙 준수가 중요합니다
3. 외래 진료 전 검사 완료를 고려하세요
4. 환자의 선호 시간대는 의료 규칙을 만족하는 범위 내에서만 반영해주세요

[금식 검사 특별 규칙]
- 금식이 필요한 검사: 환자가 오후를 원해도 반드시 오전 10시 이전으로 추천
- 금식 검사에서 환자가 "오후"를 선호한다고 해도, 의료적 이유로 오전만 가능함을 안내
- 예시 응답: "금식이 필요한 검사이므로 오전 10시 이전에만 예약이 가능합니다."
```

### 커스터마이징 가이드

## 7. 파일 간 관계

```
┌─────────────────────────────────────────────────────────────────┐
│                       데이터 참조 관계                           │
└─────────────────────────────────────────────────────────────────┘

  patient_schedule.csv                    resource_schedule.csv
  ┌──────────────────┐                   ┌──────────────────────┐
  │ 처방ID (PK)       │◄─────────────────│ 처방ID (FK)          │
  │ 환자번호         │                   │ 리소스ID             │
  │ 처방코드 (FK)    │───┐               │ 장비유형             │
  │ 예약일시         │   │               │ 실제시작시간         │
  │ 실제시작시간     │   │               │ 종료시간             │
  │ 가장빠른외래예약일자│   │               └──────────────────────┘
  └──────────────────┘   │
                         │
                         │
  constraints.csv        │
  ┌──────────────────┐   │
  │ 처방코드 (PK)    │◄──┘
  │ 처방명           │
  │ 장비유형         │────────────► resource_schedule.장비유형
  │ 소요시간         │
  │ 태그             │
  │ 규칙             │
  └──────────────────┘
```

### 참조 관계

1. **patient_schedule.처방코드** → **constraints.처방코드**
   - 모든 예약의 처방코드는 constraints에 존재해야 함

2. **resource_schedule.처방ID** → **patient_schedule.처방ID**
   - 1:1 관계 (각 예약은 하나의 리소스에 배정)

3. **constraints.장비유형** → **resource_schedule.장비유형**
   - 검사 배정 시 장비유형별 리소스 매핑

## 8. 데이터 검증 규칙

### 필수 검증

1. **처방코드 존재 여부**: patient_schedule의 모든 처방코드가 constraints에 존재
2. **장비유형 매핑**: constraints의 모든 장비유형이 유효함
3. **처방ID 일관성**: resource_schedule의 처방ID가 patient_schedule에 존재

### 형식 검증

1. **날짜 형식**: `YYYY-MM-DD` 또는 `YYYY-MM-DD HH:mm`
2. **시간 형식**: `HH:mm` (10분 단위)
3. **JSON 형식**: 태그와 규칙은 유효한 JSON 배열

### 연속 검사 검증

1. **순차 배치**: 같은 환자+예약시간+장비유형의 검사는 순차적으로 배치
2. **시간 연속성**: 이전 검사 종료 시간 = 다음 검사 시작 시간

### 규칙 검증

1. **검사간 규칙**: 같은 환자의 예약이 규칙을 위반하지 않음
   - 예: 핵의학 검사 후 1일 이내 다른 핵의학 검사 불가
